# Дашборд и устойчивость: задачи до завершения, единая статистика

Рекомендации по тому, что будет **лучше и поддерживаемее**: одна красивая и полная статистика (задачи, ошибки, база, запросы к MCP) и поведение «все задачи работают пока не завершатся», в том числе после перезапуска.

---

## 1. Задачи до завершения и после перезапуска (уже есть + что усилить)

### Что уже есть

- **Ingest:** кэш в SQLite (`ingest_cache`) — какие файлы уже проиндексированы. При новом запуске `ingest` пропускает их и обрабатывает только оставшиеся. При падении одного файла цикл не обрывается (ошибка пишется в `ingest_failed`, обработка идёт дальше).
- **Watchdog:** при ненулевом коде возврата ingest при следующем опросе снова запускает `ingest` (retry), пока не успех.
- **load-standards / load-snippets:** при запуске из watchdog или cron выполняются до конца; при падении процесса следующий запуск (по расписанию или при изменении файлов) выполнит загрузку заново.

Итого: задачи **дорабатывают** оставшуюся работу при повторном запуске за счёт кэша и повторных запусков, в том числе после перезапуска контейнера/процесса.

### Что можно усилить (опционально)

- Явный «незавершённый прогон»: при старте проверять, что в `ingest_current` или в последнем `ingest_runs` статус не `completed`, и один раз перезапустить ingest (или вывести в дашборде «последний run не завершён, будет retry»). Сейчас watchdog и так делает retry при следующем poll.
- Единая очередь задач (Celery + Redis или своя таблица в SQLite) — имеет смысл только если появятся много разных типов фоновых задач и нужна явная очередь с приоритетами; для текущего набора (ingest, load-standards, load-snippets) кэш + retry достаточно и проще в поддержке.

**Вывод:** для «все задачи работают пока не завершатся даже после перезапуска» текущая схема (кэш + retry) — **оптимальный баланс**; поддерживаемость выше, чем у отдельной очереди.

## 2. Один красивый и полный дашборд — что выбрать

Требования: одна картинка по **задачам**, **ошибкам**, **базе** (Qdrant) и **запросам к MCP**, плюс поддерживаемость.

### Вариант A: Rich (Progress + Live) в CLI — рекомендуется

- **Поддерживаемость:** одна популярная библиотека (Rich), стабильный API, много примеров и обновлений.
- **Интеграция:** все данные уже есть: `read_ingest_status()`, `read_last_ingest_run()`, `read_last_ingest_failed()`, `get_index_status()`, `get_all_collections_status()`, `read_last_snippets_run()`, маркеры `load_standards.running` / `load_snippets.running`. Новый экран — один цикл обновления (Live), который собирает эти вызовы и рисует панели.
- **Что показывать:**
  - **Задачи:** ingest (текущий/последний run: этап, done/total, ETA), load-standards / load-snippets (loading… или последний результат).
  - **Ошибки:** блок Failed из последнего ingest + при необходимости из логов.
  - **База:** коллекции Qdrant (onec_help, onec_help_memory), кол-во точек, размер.
  - **Запросы к MCP:** счётчик вызовов (см. ниже).
- **Реализация:** одна команда, например `python -m onec_help dashboard` (или `dashboard`), внутри — `rich.live.Live` + обновление раз в N секунд из тех же функций, что и для дашборда. Без нового хранилища и без веб-сервера.

Плюсы: один экран, всё в терминале, один стек (Python + Rich), легко развивать и сопровождать.

### Вариант B: Textual (TUI)

- Полноценный интерактивный TUI: вкладки, таблицы, горячие клавиши.
- Плюс: максимально «красивый» и гибкий интерфейс. Минус: больше кода и зависимостей, сложнее, чем один Live-экран с Rich.

Имеет смысл, если позже понадобятся интерактивные действия (выбор задачи, перезапуск, фильтры по ошибкам).

### Вариант C: Веб-дашборд (FastAPI/Flask + HTML)

- Отдельное приложение читает те же источники (SQLite, Qdrant API) и отдаёт страницу с графиками/таблицами.
- Плюс: красивый UI, доступ с других машин. Минус: отдельный сервис, деплой, безопасность; для «посмотреть статистику на своей машине» избыточно.

**Рекомендация:** для «красивая и полная статистика по задачам, ошибкам, базе и запросам к MCP» и максимальной поддерживаемости — **вариант A (Rich Live)**. При росте требований к интерактиву — рассматривать Textual.

## 3. Учёт запросов к MCP

Чтобы в дашборде показывать **запросы к MCP** (количество, ошибки, возможно последние вызовы), нужно считать вызовы на стороне MCP-сервера.

- **Минимум:** в памяти процесса MCP — счётчик `requests_total` и опционально `requests_errors`; при каждом вызове инструмента увеличивать. В дашборде (на том же хосте) эти цифры не увидеть без общего хранилища.
- **Вариант для одного хоста:** MCP при каждом вызове инструмента дописывает строку в файл (например `data/ingest_cache/mcp_requests.jsonl` или SQLite). CLI `dashboard` читает этот файл и показывает: всего запросов, за последнюю минуту/час, ошибки (если пишем их туда).
- **Альтернатива:** лог-файл MCP с структурированными строками (JSON), парсить его из дашборда. Менее надёжно, чем явная запись из кода.

Рекомендация: простой счётчик в файле или SQLite (одна таблица: timestamp, tool_name, success). Пишет MCP, читает дашборд — без нового сервиса, поддерживать легко.

## 4. План реализации (по шагам)

1. **Добавить зависимость Rich** в `pyproject.toml` (optional, например `[dashboard]` или в `dev`).
2. **Команда `dashboard`** в CLI: цикл `Live.update()` с интервалом 2–5 с, внутри:
   - вызовы `read_ingest_status()`, `read_last_ingest_run()`, `read_last_ingest_failed()`, `get_index_status()`, `get_all_collections_status()`, `read_last_snippets_run()`, проверка маркеров standards/snippets loading;
   - сбор блоков: Tasks (ingest + standards + snippets), Errors (failed tasks), DB (Qdrant), опционально MCP requests;
   - вывод через `rich.panel.Panel`, `rich.table.Table`, `rich.progress` при необходимости.
3. **Метрики MCP (опционально):** в `mcp_server.py` при вызове инструмента — запись в `data/ingest_cache/mcp_metrics.db` (SQLite) или в JSONL; в дашборде — чтение и отображение «Запросы к MCP: N всего, K за последний час».
4. **Документация:** в README/AGENTS.md указать запуск: `python -m onec_help dashboard` (или `make dashboard` при наличии).

Итог: **лучше и поддерживаемее** — один Rich-based дашборд в CLI, данные из существующих функций и (при необходимости) из простого счётчика запросов MCP; задачи до завершения и после перезапуска обеспечиваются уже существующим кэшем и retry.

**Реализация:** пошаговый план внедрения Rich и команды `dashboard` описан в [docs/plan-rich-dashboard.md](plan-rich-dashboard.md).
