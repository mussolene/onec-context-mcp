# Сравнение распаковщиков: «наш» (unpack_hbk) vs контейнер + TOC (read-hbk-container)

## Цель

Проверить на реальных .hbk из `/opt/1cv8` (5–10 файлов, разные платформы и версии), как отличаются по полноте результата:
- **«Наш» распаковщик** — `unpack_hbk` (цепочка 7z → zipfile → zip from offset → unzip → scan local headers → HBK container).
- **Вариант с TOC** — принудительное извлечение через `read-hbk-container` (бинарный контейнер 1С, FileStorage + PackBlock TOC).

## Набор файлов

10 файлов из трёх версий:
- **8.3.27.1719:** extui_ru, schemui_uk, basicui_lv, mapui_uk, shlang_ru, edbui_el
- **8.3.13.1513:** schemui_uk, basicui_lv
- **8.5.1.1150:** edbui_el, schemui_uk

Запуск: `PYTHONPATH=src python scripts/compare_unpackers.py` (из корня репозитория).

## Выводы по полноте

1. **Когда срабатывает «наш» распаковщик**  
   Для всех 10 файлов один из методов 1–5 (7z, zipfile, offset, unzip, scan) сработал первым. Поэтому **шаг 6 (HBK container) ни разу не выполнялся** — файл `.toc.json` при обычном `unpack` не создавался.

2. **Когда используем контейнер**  
   При явном вызове `read-hbk-container` для тех же .hbk мы получаем извлечение по формату контейнера и в **9 из 10** случаев — TOC-файл (оглавление). Исключение в тесте — extui_ru (в контейнере нет PackBlock TOC или он пустой).

3. **Полнота содержимого по объёму**  
   - **schemui_uk** (все три версии): «наш» путь дал по 2 файла (~766 байт) — по сути только фрагмент (например, только `0` и `IndexPackLookup`). Контейнер дал по 6 файлов (~24 КБ), включая формы, `__categories__` и `.toc.json`. Для таких .hbk контейнер даёт **значительно более полный** справочный материал.
   - **extui_ru**: «наш» — 5 файлов (~6 КБ), контейнер — 4 файла (~26 КБ). Разный состав (другая структура/имена), контейнер даёт больший объём контента.
   - Остальные (basicui_lv, mapui_uk, shlang_ru, edbui_el): количество файлов и размеры близки; контейнер дополнительно даёт TOC и в ряде случаев чуть другой набор/структуру файлов.

4. **Платформы и версии**  
   Поведение согласованное: 8.3.13, 8.3.27 и 8.5.1 — везде контейнер либо даёт сопоставимый результат, либо (как для schemui_uk) существенно полнее.

## Синергичный распаковщик (реализовано)

В `unpack_hbk` для файлов с расширением `.hbk` **сначала** вызывается вариант контейнера (`_try_hbk_container`): при успехе получаем полный FileStorage и `.toc.json` (оглавление). Если контейнер не подошёл или файл не в формате контейнера — выполняется прежняя цепочка: 7z → zipfile → zip from offset → unzip → scan local headers. Таким образом один вызов `unpack` даёт и полноту контента, и TOC во всех случаях, когда .hbk является бинарным контейнером 1С.

## Кодировки и «кракозябры»

- **Содержимое файлов (HTML/текст)** обрабатывается одинаково для любого способа распаковки: в `html2md` и индексаторе используется `read_file_with_encoding_fallback` (порядок: utf-8, cp1251, cp866, latin-1; переменная `HELP_FILE_ENCODING` может задать приоритет cp1251). Есть автоисправление типичных кракозябр (`_try_fix_mojibake`, детекция UTF-8, прочитанного как однобайтовая кодировка).
- **Контейнер (HBK):** имена сущностей читаются в UTF-16-LE; текст TOC из PackBlock декодируется с fallback: сначала utf-8, при ошибке — cp1251.
- **Scan local headers:** имена файлов из локальных заголовков ZIP декодируются через `_decode_filename`: utf-8, затем cp1251 (типичная кодировка 1С под Windows).
- **7z / zipfile / unzip:** имена и байты выходят «как есть» из архива; возможные искажения зависят от кодировки внутри архива и от того, как 7z/unzip интерпретируют флаг UTF-8 в ZIP. Контейнерный путь извлекает один и тот же FileStorage-ZIP, кодировки в нём единообразны.

Итого: больше всего контроля над кодировкой даёт путь контейнера (явные utf-8/cp1251 для TOC) и общий слой чтения файлов (fallback + mojibake). Scan дополнен fallback cp1251 для имён; при контейнере первым для .hbk контент чаще идёт из одного и того же ZIP, что снижает разнобой кодировок.

## Рекомендации

- Скрипт сравнения: `scripts/compare_unpackers.py`. Результаты распаковки по умолчанию пишутся в `data/compare_unpack/our/` и `data/compare_unpack/toc/` (по ключу `версия_имя.hbk`).
- При проблемах с кодировкой в уже распакованных файлах: задать `HELP_FILE_ENCODING=cp1251`, если справка точно в Windows-1251.
