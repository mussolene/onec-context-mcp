# 1С: платформа — CLI, агент сервера, создание базы

## Термины

- **Агент сервера кластера** — процесс **`ragent`**: управляет **кластером серверов 1С** (не путать с другими «агентскими» режимами ниже).
- **Конфигуратор в агентском режиме** (Designer in agent mode) — режим работы **через SSH** к **автономному серверу** (`ibsrv`): внешний SSH/SFTP-клиент выполняет команды, **аналогичные пакетным командам конфигуратора** (ограниченный набор). Описан в руководстве в связке с SSH-шлюзом автономного сервера.
- **`rac` / `ras`** — администрирование **кластера** из консоли (не автономный сервер).
- **`ibsrv`** — исполняемый файл **автономного сервера** (веб/HTTP, прямой TCP, SSH и др.; не полноценная замена сервера кластера в проде).
- **`ibcmd`** — утилита **администрирования автономного сервера и ИБ** (в т.ч. «офлайн» по каталогу данных и удалённо по **SSH**).

## Агент сервера кластера (`ragent`)

**Назначение:** центральный процесс **кластера серверов 1С** на машине, где установлен сервер приложений. Запускает и координирует рабочие процессы, участвует в регистрации кластера и обмене с другими узлами.

**Типичные элементы (ориентиры, не замена справочнику по портам в вашей инсталляции):**

- отдельные **порты** для агента, регистратора кластера и **диапазон динамических портов** для рабочих процессов — задаются при установке/в настройках службы;
- несколько версий платформы на одном хосте — **разные диапазоны портов**, чтобы кластеры не пересекались;
- запуск как **служба Windows** / **демон Linux** либо из командной строки (см. раздел *Running the server agent* в руководстве администратора клиент-сервер на kb.1ci.com).

**Практический вывод для агента Cursor:** при вопросах «почему не поднимается сервер», «занят порт», «два кластера на одной машине» — смотреть документацию по **запуску ragent** и сетевым параметрам кластера для конкретной ОС.

## Утилиты `ras` и `rac` (администрирование кластера из CLI)

- **`ras`** — сервер администрирования кластера (служба/процесс), к которому подключаются административные клиенты.
- **`rac`** — **консольный клиент** для выполнения операций администрирования (список кластеров и ИБ, сеансы, блокировки и т.д.) **через подключение к `ras`**.

**Важно:** версии **`rac` и `ras` должны соответствовать** друг другу и линейке платформы (в документации перечислены совместимые диапазоны версий).

**Типичный паттерн вызова (идея, не копировать вслепую):** указать хост/порт `ras` и подкоманду утилиты. Точный синтаксис и полный список команд — в разделе руководства администратора про **Server cluster administration software** (например [5.3.2 на kb.1ci.com](https://kb.1ci.com/1C_Enterprise_Platform/Guides/Administrator_Guides/1C_Enterprise_8.3.22_Administrator_Guide._Client_Server_Mode/5._Administration/5.3._Server_cluster_administration_software/5.3.2._Server_cluster_administration_server/)).

Полезная внешняя сводка (неофициально, но удобно как шпаргалка): репозиторий с примерами команд [salexdv/ras-rac-commands](https://github.com/salexdv/ras-rac-commands) — всё равно сверять с официальной документацией.

## Автономный сервер (`ibsrv`)

**Назначение:** компонент платформы **«автономный сервер»** — публикует ИБ для **тонкого/веб-клиента**, отладки, HTTP-мониторинга и т.д. Параметры командной строки **имеют приоритет** над одноимёнными в **файле конфигурации** (`--config=` / `-c`).

**Запуск (общий вид):**

```text
ibsrv [--parameter[=value]] ...
ibsrv --help
```

**Типовые группы возможностей (шлюзы)** — включаются/отключаются флагами; по документации 8.3.27 **по умолчанию включены** прямой TCP-шлюз, HTTP-шлюз и **SSH-шлюз**, если явно не задано иное:

| Идея | Параметры (фрагмент) |
|------|----------------------|
| Прямое подключение (аналог `/range`, `/regport` у агента) | `--enable-direct-gate` / `--disable-direct-gate`, `--direct-regport` (по умолчанию **1541**), `--direct-range` (по умолчанию **1560:1591**), `--direct-seclevel` |
| HTTP (веб-клиент и др.) | `--enable-http-gate` / `--disable-http-gate`, `--http-port` / `-p` (по умолчанию **8314**), `--http-address` / `-a`, `--http-base` / `-b` |
| **SSH** | `--enable-ssh-gate` / `--disable-ssh-gate`, `--ssh-port` (по умолчанию **1543**), `--ssh-address`, `--ssh-host-key` (путь к **приватному ключу хоста**; на Windows по умолчанию ориентир `%USERPROFILE%\.ssh\id_rsa`) |
| СУБД / файловая ИБ | `--dbms=` (MSSQLServer, PostgreSQL, IBMDB2, OracleDatabase; без параметра — **файловая**), `--db-server`, `--db-name`, `--db-user`, `--db-pwd`, `--database-path` / файловый каталог, `--request-db-pwd` / `-W` (запрос пароля СУБД из stdin) |
| Отладка | `--debug=` none \| tcp \| http \| server, `--debug-port`, `--debug-address`, … |
| Режимы ОС | Windows: `--service`; Linux: `--daemon` |
| Каталог данных сервера | `--data=` / `-d` (по умолчанию на Windows в духе `%LOCALAPPDATA%\1C\1cv8\ss-data\`) |

Полная таблица параметров — только в [Appendix 4.9 (ibsrv)](https://kb.1ci.com/1C_Enterprise_Platform/Guides/Administrator_Guides/1C_Enterprise_8.3.27_Administrator_Guide/Appendix_4._Startup_command_lines_of_system_components_and_description_of_additional_utilities/4.9._Standalone_server__ibsrv_/?language=en) для вашей версии.

## Утилита администрирования (`ibcmd`)

**Назначение:** CLI для **управления автономным сервером** и **операций с ИБ** без обязательного GUI. Входит в поставку **серверных/соответствующих компонентов** платформы (конкретный каталог установки — по документации и ОС).

**Способы указать экземпляр автономного сервера / ИБ** ([7.3.5.1](https://kb.1ci.com/1C_Enterprise_Platform/Guides/Administrator_Guides/1C_Enterprise_8.3.27_Administrator_Guide/Chapter_7.Standalone_server/7.3._Using_the_feature/7.3.5._Standalone_server_management/?language=en)):

| Режим | Параметр | Смысл |
|--------|-----------|--------|
| Локальный процесс | `--pid <PID>` | Подключение к **уже запущенному** `ibsrv` на **этой** машине; PID — номер процесса ОС. |
| Удалённо по **SSH** | `--remote ssh://<хост>:<порт>` | Пример из документации: `ssh://server1C:8282` — хост и **порт SSH-шлюза** автономного сервера (порт должен совпадать с `--ssh-port` на стороне `ibsrv`, если он нестандартный). На удалённой стороне должен быть **включён SSH-доступ** к автономному серверу. |
| «Офлайн» | `--data <каталог>` | Работа с **каталогом данных** ИБ **без** запущенного экземпляра `ibsrv`, использующего этот же каталог. Если `ibsrv` уже работает с этим `--data`, офлайн-режим для него **недоступен**. |

**Аутентификация пользователя ИБ (1С):** при необходимости доступа к ИБ — **`--user`**, **`--password`** (интерактивно или в скрипте с осторожностью). Для удалённого SSH в документе указано: сеанс может создаваться **от пользователя, прошедшего аутентификацию на SSH-шлюзе**.

**Ограничения при подключении по `--pid` или `--remote` (SSH):** недоступны, в частности: **все команды инициализации конфигурации сервера** (`ibcmd server config …`), **`ibcmd infobase create`**, **`ibcmd infobase repair`** — их выполняют в других режимах (офлайн/`--data` или до запуска сервера), см. [7.3.5.1.3–7.3.5.1.4](https://kb.1ci.com/1C_Enterprise_Platform/Guides/Administrator_Guides/1C_Enterprise_8.3.27_Administrator_Guide/Chapter_7.Standalone_server/7.3._Using_the_feature/7.3.5._Standalone_server_management/?language=en).

**Области команд (верхний уровень):** в документации и практике встречаются ветки вроде **`ibcmd server`** (конфигурация автономного сервера, в т.ч. `config init` с параметрами СУБД), **`ibcmd infobase`** (создание, загрузка/выгрузка `.cf`/`.dt`, расширения, обновление БД и т.д.), **`ibcmd session`** (сеансы). Точный список подкоманд и синтаксис — **`ibcmd --help`** и [Appendix 4.10 (ibcmd)](https://kb.1ci.com/1C_Enterprise_Platform/Guides/Administrator_Guides/1C_Enterprise_8.3.27_Administrator_Guide/Appendix_4._Startup_command_lines_of_system_components_and_description_of_additional_utilities/4.10._Standalone_server_administration_utility__ibcmd_/?language=en).

## Отладка, доступ к СУБД и инкрементальные изменения конфигурации

**Три разных смысла «SSH» — не смешивать:**

| Механизм | Назначение | Когда проще |
|----------|------------|-------------|
| **SSH-шлюз `ibsrv`** | Администрирование и агентский режим (**`ibcmd --remote`**, конфигуратор в агентском режиме, SFTP в каталог пользовательских данных) | Удалённый хост с автономным сервером; **не** заменяет доступ к порту PostgreSQL и **не** является протоколом отладчика |
| **OpenSSH port forwarding** (`ssh -L локальный_порт:хост_БД:5432 …`) | Доступ с рабочей машины к **СУБД**, если прямой порт закрыт файрволом | Типовой DevOps-приём; строка подключения 1С указывает на **проброшенный** localhost-порт |
| **VPN / прямой маршрут** | Клиент-сервер: доступ к **кластеру** и к БД по политике сети | Часто проще, чем отдельный туннель на каждый инструмент |

**Отладка кода 1С (BSL):**

- Отладчик использует **сервер отладки платформы** (отдельный контур от SSH-шлюза администрирования). У **`ibsrv`** в справке задаются **`--debug`**, **`--debug-port`**, **`--debug-address`**, **`--debug-password`** и т.д. (см. [Appendix 4.9 — ibsrv](https://kb.1ci.com/1C_Enterprise_Platform/Guides/Administrator_Guides/1C_Enterprise_8.3.27_Administrator_Guide/Appendix_4._Startup_command_lines_of_system_components_and_description_of_additional_utilities/4.9._Standalone_server__ibsrv_/?language=en)); на **клиент-сервере** — настройки по главам руководства про **отладку** и сеть между ПК разработчика и сервером (часто **VPN**).
- **Файловая ИБ:** отладка обычно **локально** (каталог `File="…"` виден ОС). «SSH к базе» в буквальном смысле для файлов — это **SSHFS / сетевой диск / копирование**, не шлюз 1С.

**Инкрементальные изменения конфигурации (выгрузка в XML), не путать с инкрементом данных в таблицах СУБД:**

- **`ibcmd config export`:** **`--base` / `-b`** — файл **ConfigDumpInfo** «относительно которого вычисляются изменения»; **`--sync`** — синхронизировать уже существующую выгрузку с конфигурацией; при несовпадении формата выгрузки — **`--force`** (полная выгрузка). См. **`ibcmd help config`** для вашей версии.
- **`ibcmd config import files`:** **`--partial`** — допускает **неполный** набор XML-файлов объектов (нужна дисциплина и понимание ссылок; иначе риск битой целостности).
- **Повседневная разработка** чаще через **EDT + Git** или **хранилище**: инкремент — в проекте и ревизиях, а `ibcmd`/`LoadCfg` — для конвейеров и администрирования.

**Где это «нормально» работает:** одна линейка **версии платформы**, согласованный **формат выгрузки**, регламент по **ConfigDumpInfo** и полным выгрузкам при смене формата; для отладки — стабильная **сеть** (или туннель/VPN) до **сервера отладки** и до **ИБ**, без подмены понятий «SSH к ibsrv» и «доступ к PostgreSQL».

Раскрыто подробнее: [1c-debug-remote-db-incremental.md](../reference/1c-debug-remote-db-incremental.md).

## SSH: подключение к автономному серверу

**1. Шлюз на стороне `ibsrv`**

- Включение/отключение: **`--enable-ssh-gate`** / **`--disable-ssh-gate`** (или секция `features` в YAML-конфиге автономного сервера).
- Сеть и ключ: **`--ssh-address`**, **`--ssh-port`** (типичное значение по умолчанию **1543**), **`--ssh-host-key`** — файл **приватного ключа хоста** для SSH.

**2. Удалённое администрирование через `ibcmd`**

- Параметр **`--remote ssh://<хост>:<порт>`** — обращение к уже запущенному автономному серверу с доступным SSH-шлюзом.

**2a. Практика (8.5.x): поднять `ibsrv` и проверить SSH**

- **Прямой шлюз и порт 1541:** если не задавать иное, автономный сервер пытается занять **1541**; при «Address already in use» процесс падает. Варианты: **`--direct-regport=<свободный>`** или **`--disable-direct-gate`**, если достаточно только SSH (и при необходимости HTTP).
- **Ключ хоста SSH:** задайте отдельный файл **`--ssh-host-key=`** (например `ssh-keygen -t rsa -b 2048 -m PEM`), не обязательно использовать личный `~/.ssh/id_rsa`.
- **`ibcmd --remote=ssh://127.0.0.1:<ssh-port> …`:** канал работает (проверено на **8.5.1** с `help`). При **первом** подключении клиент спрашивает **`[y/n]`** по отпечатку ключа хоста; в **CI без TTY** это неудобно (нужен псевдо-терминал или однократный интерактивный прогон под тем же пользователем; явного флага «trust host» в выводе `ibcmd --help` нет). После согласия узел добавляется в **внутренний** список известных узлов `ibcmd` (это не OpenSSH-`known_hosts`).
- **Операции с конфигурацией/ИБ** по SSH часто требуют **входа в информационную базу**; передача пустого пароля с pipe без TTY может давать ошибки ввода — для скриптов надёжнее **PTY** (`script`, Python `pty`, `expect`) или заранее настроенная учётка по доке версии.
- **`ibcmd --pid=`** к локальному `ibsrv` в одном из тестов на **macOS** не установил соединение (сообщение про «процесс запуска/остановки»); как запасной путь для той же машины: **`--remote`** при включённом SSH или **офлайн** **`--db-path`**, если к каталогу ИБ не держит блокировку другой `ibsrv`.

**2b. Удобство для пакетных сборок и «много потоков»**

- **Постоянно поднимать/гасить `ibsrv`** ради каждой мелкой операции обычно **медленнее и сложнее**, чем **`ibcmd config --db-path=<файловая ИБ>`** без сервера (нет SSH, нет первичного trust). Имеет смысл **долгоживущий** `ibsrv` на выделенном агенте или офлайн-режим на локальной ИБ.
- **Параллельность:** одна ИБ и один `ibsrv` всё равно упираются в блокировки; «много потоков» в CI достигается **несколькими независимыми** каталогами ИБ (и при необходимости несколькими экземплярами `ibsrv` с разными **`--data`** / портами), а не десятками `ibcmd` в один сокет к одной базе. Для **одного** большого **экспорта в XML** у `config export` в справке есть **`--threads`** — это про потоки внутри операции, не про параллельные разные задания на одной БД.
- **Внешний OpenSSH** к порту SSH-шлюза **не** даёт обычной shell-сессии: сервер разрывает сессию после рукопожатия без клиентского протокола 1С — для автоматизации ориентируйтесь на **`ibcmd --remote`**, а не на `ssh user@host`.

**3. Конфигуратор в агентском режиме через сторонний SSH/SFTP-клиент**

Если SSH для автономного сервера включён, сервер может выступать в роли **конфигуратора в агентском режиме** для **внешних** SSH/SFTP-клиентов (не только `ibcmd`). Набор команд **урезан** относительно полного пакетного режима конфигуратора; в документации перечислены группы:

- **common:** `connect-ib`, `disconnect-ib`
- **options:** `get`, `list`, `set`
- **config:** `dump-cfg`, `load-cfg`, `dump-config-to-files`, `load-config-from-files`, `update-db-cfg`, работа с расширениями, подпись и др.
- **infobase-tools:** `dump-ib`, `restore-ib`, `erase-data`, служебная информация и др.

Файлы/каталоги, создаваемые по **SFTP**, попадают в **каталог пользовательских данных** (как описано в руководстве). См. подраздел **«Connecting to a standalone server instance over SSH protocol»** в [7.3.5](https://kb.1ci.com/1C_Enterprise_Platform/Guides/Administrator_Guides/1C_Enterprise_8.3.27_Administrator_Guide/Chapter_7.Standalone_server/7.3._Using_the_feature/7.3.5._Standalone_server_management/?language=en) и перекрёстную ссылку на **Running Designer in agent mode** (Appendix 4.7 в том же руководстве).

**Важно:** этот SSH-контур относится к **автономному серверу**, а не к произвольному **кластеру `ragent`** и не заменяет **`rac`/`ras`**.

## Создание информационной базы из командной строки (`CREATEINFOBASE`)

Канонический шаблон из **Appendix 7** (1Ci KB, режим *Launch in infobase creation mode*):

```text
1cv8 CREATEINFOBASE <строка подключения> [/AddToList [<имя в списке ИБ>]] [/UseTemplate <файл шаблона>] [/Out <файл>] [/L<код языка>] [/VL<код локали>] [/O<скорость соединения>] [/DumpResult <файл>]
```

**Смысл:**

- **`CREATEINFOBASE`** — режим **создания ИБ** без интерактивного мастера (при корректной строке подключения).
- **`<строка подключения>`** — параметры доступа к ИБ в формате `ИмяПараметра=Значение;...` (файловая или клиент-серверная СУБД — см. отдельный подраздел **Infobase connection string** в том же приложении 7).
- **`/AddToList`** — добавить созданную ИБ в **список баз** запуска с указанным именем; если ключ **не** указать, ИБ в список **не** попадёт.
- **`/UseTemplate`** — создать из шаблона: файл конфигурации **`.cf`** или выгрузка ИБ **`.dt`**.
- **`/DumpResult`** — записать **код результата** (например `0` при успехе) в файл — удобно для скриптов CI/CD.
- **`/Out`** — журнал служебных сообщений в файл.

**Исполняемый файл:** на практике используют тот же бинарник, что и для толстого клиента/конфигуратора в каталоге установки платформы (`1cv8`, `1cv8c` — в зависимости от разрядности и компонентов; точное имя — в п. **7.1 Selecting executable file** руководства).

**Пример (файловая ИБ, из шаблона .cf, в список):** идея строки (пути и имя подставить свои):

```text
1cv8 CREATEINFOBASE File="C:\Bases\NewIB"; /AddToList "NewIB" /UseTemplate "C:\Templates\empty.cf" /DumpResult "C:\Temp\create_result.txt"
```

Точные имена параметров в строке подключения (`File`, `Srvr`, `Ref`, СУБД и т.д.) — **только** по документу **7.7 Infobase connection string** для вашей версии.

**Альтернатива в контуре автономного сервера:** создание и сопровождение ИБ через **`ibcmd infobase …`** (глава 7, appendix 4.10); это **отдельный** путь от **`1cv8 CREATEINFOBASE`**.

## Временная информационная база для пакетных операций (сборка epf, загрузка из XML и т.д.)

**Зачем:** многие пакетные команды конфигуратора (в т.ч. **`/LoadExternalDataProcessorOrReportFromFiles`**, работа с типами конфигурации) требуют контекст **информационной базы**: в командной строке это обычно **`/F <каталог_файловой_ИБ>`** (и при необходимости **`/N` / `/P`**). Без ИБ возможны пустой лог, ошибки или **потеря/искажение типов** (см. Appendix 7.4.8 — про выгрузку/загрузку внешних обработок и ссылки на объекты метаданных).

**Рекомендуемый паттерн — «одноразовая» лёгкая файловая ИБ:**

1. **`CREATEINFOBASE`** — создать **новую пустую или минимальную** файловую ИБ в каталоге вроде `/tmp/onec-build-<id>/ib` или `…\Temp\onec-build-<id>\ib`.
2. Шаблон: **`/UseTemplate <минимальный.cf>`** или **`.dt`** — взять **небольшой** шаблон (пустая/учебная конфигурация подходящей версии платформы). **Не** использовать для этого тяжёлую копию продуктивной базы: долго, лишний диск, риск случайного доступа к данным.
3. **`/AddToList`** — по желанию: **`/AddToList "ИмяВСписке"`** добавляет запись в список баз пользователя (**`ibases.v8i`**). Для чисто автоматических пайплайнов **удобнее не добавлять** (без `/AddToList` ИБ в список **не** попадает — см. описание ключа в приложении 7).
4. Запуск **`CONFIG`** с **`/F`** на этот каталог и нужной пакетной командой (сборка `.epf`, загрузка конфигурации и т.д.).
5. **После успеха:**
   - **удалить** каталог файловой ИБ и при необходимости запись из списка баз (если добавляли), либо
   - **оставить** ИБ в известном каталоге для **повторного использования** в CI/локальных сборках (ускоряет прогоны; следите за версией платформы и шаблона).

**Список баз `ibases.v8i` как источник пути к ИБ (опционально):**

- Файл **текстовый**, формат с секциями **`[ИмяБазы]`**, внутри строка подключения вида **`Connect=File="…";`** или клиент-серверный вариант. Имя секции — отображаемое имя ИБ в списке.
- Путь к файлу зависит от ОС и варианта запуска (тонкий/толстый клиент, версия платформы); его можно взять из настроек списка информационных баз или найти в профиле пользователя.
- **Для временных сборок** не ориентируйтесь на «боевую» базу из списка, если она **большая** или содержит чувствительные данные: лучше отдельная **лёгкая** ИБ, созданная **`CREATEINFOBASE`** под задачу.

**Скрипт в репозитории:** `tools/1c/build_metadata_export_epf.sh` поддерживает опции автосоздания временной ИБ (`ONEC_AUTO_TEMP_IB`, `ONEC_EMPTY_CF`, `ONEC_REMOVE_TEMP_IB` — см. комментарии в файле).

## Другие режимы командной строки (обзор)

Ниже — **типовые** режимы и ключи, которые чаще всего нужны в автоматизации. Это **не исчерпывающий** список.

| Режим / идея | Назначение (кратко) |
|--------------|---------------------|
| **`ENTERPRISE`** | Запуск **режима 1С:Предприятие** (прикладное решение) с ключами `/F`, `/S`, `/N`, `/P`, `/UC` и др. |
| **`CONFIG`** | Запуск **конфигуратора**; для пакетных операций — `/DumpCfg`, `/LoadCfg`, `/UpdateDBCfg`, `/Visible` и т.д. (см. подразделы **Running Designer in batch mode** в приложении 7) |
| **`CREATEINFOBASE`** | Создание ИБ (см. выше) |
| **`/F <каталог>`** | Файловая информационная база (каталог) |
| **`/S <сервер\имяИБ>`** | Клиент-сервер: кластер и имя ИБ в списке кластера |
| **`/N`, `/P`** | Имя пользователя и пароль |
| **`/Out <файл>`** | Вывод служебных сообщений |
| **`/DisableStartupMessages`** | Подавить часть стартовых окон (актуальность — по документации версии) |
| **`/Execute`** | Запуск внешней обработки/отчёта `.epf`/`.erf` (механика и ограничения — в документации) |
| **`/DumpCfg` / `/LoadCfg`** | Выгрузка/загрузка конфигурации (и расширений — с ключами вроде **`-Extension`**, если применимо) |

Для **тонкого клиента**, **веб-клиента**, **режима отладки**, **мобильного** и специализированных ключей — смотреть соответствующие подразделы **Appendix 7** и главу про нужный компонент в том же руководстве.

**Замечание по версиям (внешняя обработка из XML):** в документации встречается формулировка «корневой **каталог**» выгрузки; на практике **8.5.x** в журнале конфигуратора указывает «корневой **файл**» — для иерархической выгрузки передавайте путь к **`MetadataExport.xml`** (или иной корневой файл выгрузки), если пакетный запуск с каталогом не срабатывает. Сверяйте с текстом в **`/Out`** для вашей сборки платформы.

## «Headless» и пакетный конфигуратор (`1cv8 CONFIG`)

**Важно:** у платформы **нет отдельного полностью консольного** конфигуратора без графического стека. Пакетный режим означает **отсутствие интерактива** и скрытие окон ключами вроде **`/Visible false`**, **`/DisableStartupMessages`**, при необходимости **`/DisableStartupDialogs`** — но процесс по-прежнему может требовать **графическую подсистему** (GTK/Qt и т.д., зависит от сборки).

| Среда | Типичная практика |
|-------|-------------------|
| **Linux (CI, сервер без монитора)** | Поднять виртуальный дисплей (**Xvfb**), выставить **`export DISPLAY=:99`** (или `:1`), запускать **`1cv8`** под этим `DISPLAY`. Дополнительно: нужные библиотеки GTK/WebKit, шрифты — иначе возможны падения; см. обсуждения на mista/infostart и статьи по администрированию Linux. |
| **macOS** | Часто достаточно **залогиненного пользователя** с GUI-сессией; из **SSH без сессии** или **песочниц** сборка может падать или зависать. Это ограничение окружения, а не только ключей командной строки. |
| **Windows** | Пакетный запуск из **служб** / **RDP отключён** может требовать отдельной настройки интерактивной станции или запуска под пользовательской сессией. |

**Про `/Visible false`:** в ряде версий сообщалось о регрессиях / изменении поведения; при сбоях в CI сверяйте **приложение 7** и **заметки к вашей 8.3.xx / 8.5.xx** и пробуйте сочетание с **`/DisableStartupDialogs`** (только после проверки по доке).

**Итог:** «headless» для **`LoadExternalDataProcessorOrReportFromFiles`** = **пакетный CONFIG + временная ИБ + /Out** + на Linux **Xvfb + DISPLAY**, а не один волшебный флаг.

## Сводка: как получить `.epf` / `.erf` из исходников (разные пути)

| Способ | Вход | Назначение | Headless / CI |
|--------|------|------------|----------------|
| **`1cv8 CONFIG`** + **`/LoadExternalDataProcessorOrReportFromFiles`** | Иерархическая (или линейная) выгрузка в XML, корень часто **`…/MetadataExport.xml`** | **Сборка бинарного `.epf`/`.erf`** из файлов выгрузки конфигуратора | Возможен при **Xvfb+DISPLAY** (Linux) или GUI-сессии (macOS); см. раздел выше. |
| **SSH к `ibsrv`** + команда агента **`load-external-data-processor-or-report-from-files`** | `--file=` корневой XML, `--ext-file=` выходной `.epf` | То же по смыслу, что пакетный CONFIG, но **через шлюз SSH** автономного сервера | Подходит для сервера, где **`ibsrv`** уже запущен с **SSH-gate**; клиент — протокол 1С, не произвольный shell. |
| **`ibcmd infobase config load`** (и аналоги **`load` с путём к файлу**) | Файл **`.cf` / `.cfe` / уже готовый `.epf` / `.erf`** | **Загрузка в ИБ** конфигурации или **готового** внешнего отчёта/обработки, **не** компиляция дерева XML выгрузки в `.epf` | Да, **`ibcmd`** удобнее для headless-операций **с ИБ**, но **не заменяет** шаг «XML → epf». |
| **`ibcmd config`**, **`export` / `import --out=`** | Для внешней обработки: **`export --file=*.epf <каталог>`** (epf → дерево XML), **`import --out=*.epf <каталог>`** (дерево XML → epf); нужен **`--db-path=<файловая ИБ>`** (контекст утилиты) | **Сборка/разбор `.epf`/`.erf`** из иерархической выгрузки **без** пакетного `CONFIG` | Удобно в headless; **без** постоянного `ibsrv`, если к ИБ нет конкурирующей блокировки |
| **1C:EDT** + **ring / сборка проекта** | Проект EDT | Сборка поставки **из модели EDT**; исходники **другого форма**, чем выгрузка «в файлы» толстого конфигуратора | Обычно **CLI** в CI; миграция с XML-дерева MetadataExport — **отдельная** задача. |

**Проверка по справке `ibcmd` 8.5:** в ветке **`infobase` → `config` → `load`** путь трактуется как файл **конфигурации, расширения, внешней обработки или отчёта** — то есть **уже бинарный/готовый файл**, а не каталог XML-выгрузки для *сборки* epf.

**`ibsrv`** не заменяет шаг «XML → epf»: он даёт среду (**HTTP / прямой доступ / SSH**). Саму сборку **`.epf` из файлов** на практике проще делать **`ibcmd config import --out=…`** офлайн или **`1cv8 CONFIG` / агент по SSH**; по SSH тот же класс операций тянет за собой **trust ключа** и **аутентификацию ИБ**.

## Как работать агенту Cursor

1. **Уточнять версию платформы** и ОС — от этого зависят состав параметров и имена исполняемых файлов.
2. **Не выдумывать редкие ключи** — при сомнении открыть **Приложение 7** руководства администратора для этой версии (1Ci KB или ИТС).
3. Для **пакетной сборки** (`.epf`/`.erf` из XML, сценарии с разрешением типов): предусмотреть **`/F`** к **файловой ИБ**; при отсутствии подходящей — **`CREATEINFOBASE`** во **временный каталог** с **лёгким** шаблоном, без **`/AddToList`** если не нужен список баз; после работы — **удалить** ИБ или **сохранить** для повторных сборок осознанно.
4. Разделять уровни:
   - запуск **клиента/конфигуратора** (`1cv8` + режим `ENTERPRISE` / `CONFIG` / `CREATEINFOBASE` …);
   - **кластер сервера** (**`ragent`**, рабочие процессы, порты);
   - **администрирование кластера** (**`rac` → `ras`**);
   - **автономный сервер** (**`ibsrv`**) и **`ibcmd`** (в т.ч. **`--remote ssh://…`** и **офлайн `--data`**).
5. Не смешивать: **SSH-шлюз автономного сервера** / **конфигуратор в агентском режиме** — это про **`ibsrv`**, а не про **`ragent`**.
6. Для **скриптов** с классическим запуском использовать **`/DumpResult`** и **`/Out`**; для конвейеров с автономным сервером — **`ibcmd`** и коды возврата по документации.
7. Для **headless Linux CI** при сборке epf из XML закладывать **Xvfb + `DISPLAY`**, а **`ibcmd`** использовать для шагов с **ИБ** (создание/загрузка `.cf`), а не как замену **`LoadExternalDataProcessorOrReportFromFiles`**.
