# Анализ: полнота знаний, скорость ответа и scroll-операции MCP

## 1. Корректность и полнота знаний

| Аспект | Состояние | Комментарий |
|--------|-----------|-------------|
| **Семантический поиск** | Ок | `search_index` → один vector query (points/query). Результаты по релевантности. При отключённом embedding — fallback на deterministic, релевантность ниже. |
| **Ключевой поиск** | Ок | При включённом BM25 (по умолчанию при ingest) — один sparse query, без scroll. При отсутствии BM25/vocab — fallback: scroll по коллекции с фильтром по `keywords` или substring по title/text (много scroll при большой коллекции). |
| **Гибрид (get_1c_code_answer, search_with_content)** | Ок | RRF объединяет 1 семантический запрос + до 8 ключевых запросов (по токенам из запроса). При BM25 — всё через query, без scroll. |
| **Чтение темы по path** | Ок | `get_topic_from_index`: сначала scroll с фильтром по `path` (limit=1) — один быстрый scroll; fallback scroll(limit=200) только если по фильтру не нашли. |
| **Версии/языки в статусе** | Ок | Вывод версий и языков строится по выборке точек (scroll), см. раздел про scroll. |

**Итог:** Полнота и корректность данных при наличии индекса и BM25 в норме. Слабое место — семантика при работе без внешнего embedding (deterministic даёт худший порядок результатов).

---

## 2. Скорость ответа

| Операция | Запросы к Qdrant | Оценка |
|----------|-------------------|--------|
| **get_1c_help_index_status** | 1× get_collection + 1× scroll(500) | ~250 KB по сети, ~0.1–0.2 с |
| **search_1c_help** | 1× points/query (vector) | Быстро |
| **search_1c_help_keyword** (BM25) | 1× query (sparse) | Быстро |
| **search_1c_help_keyword** (fallback) | N× scroll по 500 до набора limit | Медленно при N ≫ 1 |
| **get_1c_code_answer** | 1 semantic + до 8 keyword + 1 memory query + до limit× get_topic | При BM25 — несколько query; get_topic — 1 scroll с фильтром на тему (или диск). |
| **list_1c_help_titles(limit=100)** | Scroll батчами (например 500) до 100 уникальных path | 1–2 scroll при пустом prefix, больше при жёстком path_prefix |
| **get_1c_help_topic** | 0 (диск) или 1 scroll с фильтром (limit=1) | Быстро |

Главный по объёму ответа запрос в логе — **scroll ~262 KB** для `get_index_status` (500 точек с полным payload). Остальные scroll — либо с фильтром и маленьким limit, либо батчи для list_*.

---

## 3. Количество scroll-операций

### Где используется scroll

| Функция | Назначение | Размер батча / лимит |
|---------|------------|------------------------|
| **get_index_status** | Выборка версий/языков | 1 scroll, limit=500 |
| **list_index_titles** | Список (path, title) по path_prefix | Батчи по min(500, …), до limit записей |
| **list_index_nav_items** | path, title, breadcrumb для навигации | Батчи по 500, до 5000 (не вызывается из MCP) |
| **search_index_keyword** (fallback) | Поиск по keywords/substring без BM25 | Батчи по 500 до набора limit |
| **get_topic_from_index** | Тема по path | 1 scroll с фильтром limit=1 или fallback 200 |
| **get_topic_metadata** | Метаданные по path | 1 scroll с фильтром limit=1 или fallback 100 |
| **get_1c_help_related** | Связанные по path | Scroll с фильтром по 50 |
| **compare_1c_help** | Две версии темы | Чтение через get_topic (scroll с фильтром) |
| **add_bm25_to_collection** | Миграция с BM25 | Полный scroll коллекции (админская операция) |

### Типичный сценарий (по логу)

- **get_1c_help_index_status**: 1 scroll (500 точек) → ~262 KB.
- **list_1c_help_titles(limit=5, path_prefix="8.3")**: 1–2 scroll (батч до 500, отбор по prefix).
- **get_1c_code_answer** (при BM25): 0 scroll для поиска; по 1 scroll с фильтром на каждую из выбранных тем (часто 1–3).
- **search_1c_help_keyword** при включённом BM25: 0 scroll.

Итого при нормальной конфигурации (BM25 включён): основной «тяжёлый» scroll — один раз на вызов **get_1c_help_index_status**.

---

## 4. Что можно улучшить

### 4.1. get_index_status — уменьшить объём scroll (рекомендуется)

- **Было:** один scroll с `limit=500` для выборки версий/языков (~262 KB при 117k точках).
- **Сделано:** по умолчанию `sample_size=100`; задаётся через env **INDEX_STATUS_SAMPLE_SIZE** (50–2000). 100 точек достаточно для репрезентативной выборки версий/языков.
  - Опционально: кэш статуса на 30–60 с, чтобы при частых вызовах не повторять scroll. Риск — устаревший прогресс ingest; при необходимости кэш можно отключать или делать только для «версии/языки», а прогресс всегда читать из ingest state.

**Эффект:** меньше трафика и времени на каждый вызов get_1c_help_index_status.

### 4.2. list_index_titles / list_index_nav_items

- Scroll неизбежен: нет вторичного индекса «по path» для листинга; Qdrant отдаёт точки батчами.
- Можно уменьшить объём payload: запрашивать только нужные поля (path, title [и breadcrumb для nav]), если клиент/API поддерживает выбор полей — тогда меньше данных по сети при том же числе scroll.

### 4.3. search_index_keyword без BM25

- При отсутствии BM25 или vocab срабатывает fallback с множественным scroll по коллекции — при 117k точках это дорого.
- **Рекомендация:** не менять логику, но в проде держать BM25 включённым (ingest с BM25 по умолчанию; при инкрементальном ingest — потом выполнить `add-bm25`). В доке явно указать, что без BM25 ключевой поиск медленный.

### 4.4. Семантика без внешнего embedding

- При отключённом API используется deterministic-вектор; порядок результатов хуже, чем с нормальным эмбеддером.
- Менять ничего не обязательно: fallback корректен, качество улучшается только включением embedding API.

---

## 5. Вывод

- **Корректность и полнота:** при наличии индекса и BM25 — в порядке; при отключённом embedding качество семантики ниже, но механизм корректен.
- **Скорость:** упирается в один «тяжёлый» scroll в **get_index_status** (500 точек) и при необходимости в list_* и keyword-fallback.
- **Scroll:** основное улучшение — уменьшить объём scroll в get_index_status (например, sample_size=100 или настраиваемый размер). Остальное при текущей архитектуре приемлемо; BM25 в проде желателен, чтобы не попадать в keyword fallback со множественным scroll.

Менять общую архитектуру (отказ от scroll для статуса, введение отдельного хранилища метаданных) при текущих объёмах не обязательно — точечная оптимизация get_index_status достаточна.
