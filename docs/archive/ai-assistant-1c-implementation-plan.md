# План: AI-помощник 1С с metadata graph

Дорожная карта реализации AI-ассистента для 1С с учётом HBK, Qdrant, BSL LS, Cursor, metadata graph и версионности конфигураций 1С.

---

## 0. Аудит и границы изменений

### Цель аудита
- Зафиксировать, какие части системы стабильны и **не трогаем**: ingest HBK, формат коллекции `onec_help`, поведение существующих MCP-инструментов (`search_1c_help`, `get_1c_code_answer` и т.д.).
- Определить **зоны расширения**: новые модули, новые Qdrant-коллекции, новые MCP-инструменты.

### Решения по границам
- **Не меняем:**
  - Формат коллекции `onec_help`, логику `indexer.build_index` для справки, `ingest.py`, существующие CLI-команды и их семантику.
  - Сигнатуры и поведение уже работающих MCP-инструментов.
- **Добавляем:**
  - Модуль `metadata_graph` (или аналог) — только новые коллекции и API.
  - Модуль `context_builder` — композиция существующих и новых источников.
  - Новые MCP-инструменты: `search_1c_metadata`, `get_1c_metadata_object`, `get_1c_context_bundle`.
- **Результат аудита** (зафиксировать в коде/доках при реализации):
  - Список новых модулей и их ответственность.
  - Список новых Qdrant-коллекций и схема payload.
  - Явное описание: текущий код продолжает работать без изменений; новый функционал по умолчанию либо включён с разумными дефолтами, либо при отсутствии данных возвращает понятную ошибку «функционал не настроен».

## 1. Версионность 1С и metadata graph

1С — **версионная платформа**: конфигурации сильно меняются от версии к версии (8.2, 8.3.x, разные релизы типовых конфигураций). Metadata graph должен это учитывать.

### Требования к версионности
- **В payload узлов графа хранить:**
  - `platform_version` — версия платформы (например `8.3.24`, `8.3.25`).
  - `config_version` или `config_release` — версия/релиз конфигурации, если известна (например из EDT, выгрузки или настроек проекта).
- **Идентификация объектов:**
  - Уникальный id узла должен включать версию (или граф строится отдельно на конфигурацию/версию), чтобы не смешивать объекты из разных версий в одном наборе.
- **Варианты хранения:**
  - **Вариант A:** Одна коллекция узлов с обязательным фильтром по `platform_version` (и при необходимости по `config_version`) при поиске; при `metadata-graph-build` передавать версию из env или из метаданных выгрузки.
  - **Вариант B:** Отдельные коллекции/namespace на версию (например `onec_metadata_v83_24`), если ожидается много параллельных конфигураций.
- **Рекомендация для первого этапа:** Вариант A — одна коллекция, обязательные поля `platform_version` и опционально `config_version` в payload; все запросы к metadata graph принимают параметр `platform_version` (или берут из конфига/контекста сессии), чтобы поиск и контекст всегда были привязаны к нужной версии.
- **В инструкции по экспорту метаданных** явно указать: при смене версии платформы или конфигурации пересобирать граф или добавлять данные с новой версией и фильтровать по ней при использовании.

## 2. Пошаговый план (с TDD и без заглушек)

### 2.1. Шаг 0. Мини-аудит
- Просмотреть ключевые места: `ingest.py`, `indexer.py`, `memory.py`, `mcp_server.py`, `cli.py`, `watchdog.py` — зафиксировать, что не меняем.
- Краткая заметка в `docs/` («Границы изменений AI-помощника 1С»).
- Запуск текущих тестов (pytest, ruff) как страховка статуса quo.

### 2.2. Шаг 1. Схема metadata graph (включая версионность)
- Типы узлов: Document, Catalog, Register, Enum, Form, Attribute, TablePart, Command и т.п.
- Типы связей: `uses`, `writes_to`, `reads_from`, `has_form`, `belongs_to_subsystem`, `opens_form`, `calls_module` и т.д.
- Payload узла: `id`, `type`, `name`, `full_name`, `module_path`, `subsystem`, `platform_version`, `config_version` (опц.), `short_description`, текст для эмбеддинга (опц.).
- Payload связи: `from_id`, `to_id`, `relation_type`, при необходимости `platform_version`.
- **TDD:** Тесты на чистые структуры (in-memory граф, сериализация в payload и обратно).

### 2.3. Шаг 2. Реализация `metadata_graph` (MVP, без заглушек)
- Подключение к Qdrant через существующую конфигурацию.
- `build_metadata_graph(source_dir, platform_version, config_version=None)` — парсинг выгрузки, создание узлов/рёбер с версией, запись в Qdrant.
- `search_metadata_by_name(query, type_filter, platform_version)` — поиск с фильтром по версии.
- `get_metadata_object(id)` — объект + связи (id уже привязан к версии или в payload есть версия).
- **Без заглушек:** при отсутствии данных — пустой результат или явная ошибка «граф не настроен», не «TODO».
- **TDD:** Фикстуры с минимальной тестовой конфигурацией (1 Документ, 1 Справочник, 1 Регистр); тесты на build и search с учётом `platform_version`.

### 2.4. Шаг 3. CLI и опционально watchdog
- Команда `metadata-graph-build` с аргументами `--source-dir`, `--platform-version` (обязательно), `--config-version` (опц.), `--reset`.
- Env: например `ONEC_CONFIG_SOURCE_DIR`, `ONEC_PLATFORM_VERSION` для дефолтов.
- Watchdog (второй итерацией): при изменении каталога конфигурации — пересборка графа с сохранением версии.
- **TDD:** Smoke-тесты CLI (парсинг аргументов, вызов build с тестовым каталогом).
- **Юзабилити:** Если source-dir или platform-version не заданы — понятное сообщение, без падений; MCP при отсутствии графа возвращает «функционал не включён».

### 2.5. Шаг 4. Context builder
- Вход (DTO): `file_uri`, `symbol_name`, `diagnostics`, `call_graph` (опц.), `config_object_name`, **`platform_version`** (для metadata).
- Внутри: вызовы `indexer.search_hybrid`, `MemoryStore.search_long`, `metadata_graph.search_metadata_by_name(..., platform_version)` и `get_metadata_object`.
- Выход: JSON с блоками `help_topics`, `snippets_and_standards`, `metadata_objects`.
- **TDD:** Юнит-тесты с моками indexer, memory, metadata_graph; проверка сборки и ранжирования.

### 2.6. Шаг 5. MCP-инструменты (только рабочие)
- `search_1c_metadata(query, type_filter, platform_version)` — вызов `metadata_graph.search_metadata_by_name`.
- `get_1c_metadata_object(id)` — вызов `metadata_graph.get_metadata_object`.
- `get_1c_context_bundle(dto)` — вызов `context_builder`; в DTO передаётся `platform_version` для metadata.
- Все инструменты при неинициализированном графе возвращают явную ошибку; без заглушек и TODO.
- **TDD:** Тесты вызовов инструментов (mock нижнего слоя), проверка формата ответа и обработки ошибок.

### 2.7. Шаг 6. Юзабилити, видимость, простой запуск
- Ответы MCP структурированы, сообщения об ошибках понятные.
- Dashboard: новые коллекции в `get_all_collections_status`, статус последнего `metadata-graph-build` (через Redis по образцу ingest).
- Запуск по умолчанию: пользователь задаёт путь к выгрузке и версию платформы → один раз `metadata-graph-build` → остальное работает.

## 3. Готовый промпт для AI-помощника 1С

Использовать как есть в Cursor rule / MCP-подсказке (без черновиков).

```text
Ты — AI-помощник по разработке на 1С (BSL) в IDE.

Твои источники знаний:
1) Справка 1С и связанные сниппеты, проиндексированные через сервис 1c_help (HBK → Markdown → Qdrant): инструменты search_1c_help, search_1c_help_keyword, get_1c_help_topic, get_1c_code_answer.
2) Граф метаданных конфигурации 1С (Документы, Справочники, Регистры, Перечисления, Формы, Табличные части, Реквизиты, команды и связи), привязанный к версии платформы и конфигурации: инструменты search_1c_metadata, get_1c_metadata_object. Учитывай, что структура конфигурации зависит от версии 1С.
3) Информация о текущем проекте (BSL LS, контекст IDE): текущий модуль, позиция курсора, имя процедуры/функции, диагностики, граф вызовов; сниппеты и стандарты команды из памяти (snippets, standards).

Цели:
- Давать точные и практические советы по коду 1С и объектам конфигурации.
- Учитывать реальную структуру конфигурации и версию платформы.
- Делать ответы компактными, читабельными и пригодными к вставке в код.

Порядок работы:
1. Понимание контекста — проанализировать входной контекст от IDE (файл, позиция, символ, диагностики, объект конфигурации, при необходимости версию платформы).
2. Сбор контекста — использовать get_1c_context_bundle (темы справки, сниппеты/стандарты, объекты metadata graph с учётом версии).
3. Формирование ответа — кратко описать задачу и объекты; предложить основной код, адаптированный под контекст и версию; при необходимости альтернативы и ссылки на справку.

Требования:
- Не выдумывать объекты и методы: опираться на metadata graph и справку 1С; учитывать версионность конфигурации.
- Учитывать диагностики BSL LS; соблюдать стандарты кода; минимизировать шаги от ответа до рабочего кода.

Формат ответа: Контекст (1–3 предложения) → Как устроены объекты (если важно) → Решение (код) → Альтернативы (опц.) → Ссылки на справку (опц.). Без избыточных пояснений.
```

## 4. Получение метаданных конфигурации 1С

### 4.1. Практический путь (выгрузка конфигурации)
1. **Через 1С:EDT:** создать/подключить проект конфигурации → синхронизация с хранилищем/базой → получить файловую структуру (XML и каталоги объектов).
2. **Или через конфигуратор:** выгрузить конфигурацию в файл (.cf) и при необходимости конвертировать в файловую структуру (внешние утилиты).
3. **Источник правды:** один каталог EDT-проекта, синхронизируемый с рабочей базой.
4. **Окружение:** переменные `ONEC_CONFIG_SOURCE_DIR` (путь к корню выгрузки), `ONEC_PLATFORM_VERSION` (например 8.3.24).

### 4.2. Скрипт/утилита (целевое поведение при реализации)
- **Команда:** `python -m onec_help metadata-graph-build --source-dir /path/to/edt/project --platform-version 8.3.24 [--config-version 2.0] [--reset]`
- **Действия:**
  - Рекурсивный обход каталога; чтение XML описаний объектов (Documents, Catalogs, Registers, Enums, формы, реквизиты, табличные части).
  - Извлечение связей (ссылки между объектами, при возможности — из модулей).
  - Сбор графа с полями `platform_version` и при наличии `config_version`.
  - Запись в Qdrant; прогресс/статус в Redis для dashboard.
- **Инструкция для пользователя:**
  1. Выгрузить конфигурацию в файловый вид (EDT), указать путь и версию платформы в env или аргументах.
  2. Запустить `metadata-graph-build`.
  3. После этого search_1c_metadata / get_1c_metadata_object и get_1c_context_bundle возвращают данные с учётом выбранной версии.

### 4.3. Версионность при экспорте
- В инструкции явно указать: при смене версии платформы или релиза конфигурации нужно пересобрать граф с новыми `--platform-version` / `--config-version` или пополнять индекс с новой версией и фильтровать по ней при запросах.

## 6. Модульность и поддерживаемость

- Чёткие границы: `metadata_graph` — только хранение и поиск по конфигурации (с версией); `context_builder` — только композиция источников; MCP — тонкие обёртки.
- Слабая связь с остальной системой: общая инфраструктура (Qdrant, Redis, embedding), но логика изолирована; при необходимости модуль metadata можно вынести в отдельный пакет.
- Схема payload и версионность зафиксированы в коде и в этой документации для упрощения расширения типов объектов и связей.
