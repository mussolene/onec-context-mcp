# План доработки веб-просмотрщика справки и кросс-ссылок

## Что уже сделано (текущий коммит)

### 1. Отзывчивость при клике по меню
- **Индикатор загрузки:** при выборе пункта в дереве сразу показывается «Загрузка…» в области контента, затем подставляется контент. Раньше экран оставался пустым до ответа сервера — создавалось впечатление, что «ничего не происходит».
- **Очистка дерева:** перед построением дерева из `/api/nav` контейнер `#tree` очищается (`treeEl.innerHTML = ''`), чтобы при повторных вызовах не дублировались узлы.
- **Обработка ошибок:** при ошибке `fetch` (не OK или `data.error`) в области контента показывается сообщение об ошибке вместо пустого экрана.

### 2. Кросс-ссылки в контенте из индекса (Markdown)
- В `tree._rewrite_content_links()` учёт текущей страницы: относительные ссылки в Markdown на соседнюю тему разрешаются относительно текущей страницы; для темы `8.2.19.130/1cv8_ru/MessageWindow` ссылка `OtherTopic` превращается в `/content/8.2.19.130/1cv8_ru/OtherTopic`, а не в `/content/OtherTopic`.
- Контент из Qdrant (Markdown → HTML) передаётся в `_rewrite_content_links(html, base, current_file)` с соответствующим `current_file`/путём, поэтому ссылки внутри статей из индекса ведут на корректные пути.

### 3. Что уже было и остаётся
- **Контент с диска (HTML):** ссылки переписываются в `tree._rewrite_content_links()` (v8help и относительные пути → `/content/...`).
- **Блок «См. также»:** берётся из `outgoing_links` в payload Qdrant; в индексаторе для `resolved_path` уже подставляется `path_prefix`, так что ссылки ведут в ту же версию/платформу.
- **fixLinks() во фронтенде:** для всех `<a href="/content/...">` в теле статьи вешается `onclick` → `loadContent(path)`.

## Что проверить вручную (или через Playwright)

1. **Скорость:** задержка при клике — это время ответа `/content/<path>?meta=1` (чтение файла или Qdrant + Markdown→HTML). Если база большая, можно:
   - кэшировать ответы по path в памяти (TTL или размер ограничить);
   - отдавать контент без meta при первом запросе и подгружать breadcrumb/outgoing_links отдельно (два запроса, но быстрее первый рендер).
2. **Клики по дереву:** убедиться, что клик по названию файла (не только по иконке) открывает контент и что активный пункт подсвечивается.
3. **Кросс-ссылки:**
   - в статье из **файла** (HTML) — ссылки вида `/content/...` должны открывать целевую тему;
   - в статье из **индекса** (Markdown) — относительные ссылки должны вести в ту же версию/каталог (логика в `tree._rewrite_content_links()`);
   - блок «См. также» — каждая ссылка должна открывать соответствующую тему (path = `resolved_path` из payload).
4. **Поиск в шапке:** поиск по контенту (`/api/search`) и переход по результату в список статей — убедиться, что открывается нужная страница и ссылки внутри неё работают.

## Возможные следующие шаги

| Задача | Описание |
|--------|----------|
| Кэш контента | In-memory кэш для `get_content` по path (с ограничением размера/TTL), чтобы повторные переходы были мгновенными. |
| Ленивая подгрузка meta | Первый запрос — только HTML контент; второй — breadcrumb + outgoing_links для блока «См. также» и навигации. |
| Playwright-тесты | Сценарии: загрузка главной, клик по пункту дерева, клик по ссылке в контенте, клик по «См. также». |
| Человекочитаемые названия в дереве | Убедиться, что в индексе для навигации заполнены `title`/`breadcrumb`; иначе в дереве остаются slug (например `MessageWindow`). При необходимости дообогащать из `__categories__` или первого заголовка в контенте. |

## Файлы изменений

- `templates/index.html`: индикатор загрузки, очистка `#tree`, обработка ошибок fetch в `loadContent`.
- `src/onec_help/tree.py`: `_rewrite_content_links(content, base, current_file)` — переписывание ссылок в контенте (v8help и относительные пути → `/content/...`) с учётом текущего пути при отдаче контента из индекса.
