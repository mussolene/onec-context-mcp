# Выводы для агента: достаточно ли MCP 1c-help для рефакторинга

Краткий разбор: что даёт MCP 1c-help, чего не хватает, и как агенту понимать «что, как и где» при рефакторинге кода 1С.

---

## 1. Что даёт MCP 1c-help (справка + память)

| Инструмент | Для рефакторинга даёт |
|------------|------------------------|
| **get_1c_code_answer** | Примеры кода, типовое использование API, сниппеты из памяти. Ответ: «как правильно писать» с учётом справки. |
| **search_1c_help_keyword** | Точный поиск по имени API (Тип.Метод). Пути топиков для последующего чтения. |
| **get_1c_help_topic** | Полный текст топика по `topic_path` (Markdown). Семантика API, ограничения, версии. |
| **get_1c_function_info** | Описание, синтаксис, параметры функции/метода. Сигнатура и контракт — что менять можно, что нельзя. |
| **get_form_metadata** | Атрибуты и команды формы из Form.xml. Контекст формы при рефакторинге модуля формы. |
| **get_module_info** | Тип модуля (FormModule, ObjectModule и т.д.) и контекст по пути к файлу. Понимание «где я нахожусь». |
| **compare_1c_help** | Отличия топика между версиями платформы. Нужно при переносе кода между версиями. |
| **save_1c_snippet** | Сохранение удачного варианта кода в память — улучшает последующие ответы get_1c_code_answer. |

Итого по 1c-help: агент получает **что такое API** (справка, сигнатуры) и **как им пользоваться** (примеры, подводные камни). **Где в кодовой базе** что вызывается — 1c-help не знает.

---

## 2. Что даёт lsp-bsl-bridge (проект и диагностика)

| Инструмент | Для рефакторинга даёт |
|------------|------------------------|
| **project_analysis** | Поиск символов, файлов, текста по проекту. «Где объявлено/используется». |
| **symbol_explore** | Детали по символу (тип, определения, ссылки). |
| **call_graph** | Граф вызовов. Обязательно перед рефакторингом — оценка влияния изменений. |
| **document_diagnostics** | Ошибки, предупреждения, стиль BSL LS по файлу. Проверка после каждой правки. |
| **code_actions** | Быстрые исправления (ограниченный набор). |
| **prepare_rename / rename** | Безопасное переименование после анализа call_graph. |
| **did_change_watched_files** | Уведомление LSP об изменениях после batch-правок. |

Итого по lsp-bsl-bridge: агент получает **где что лежит и кто кого вызывает**, а также **что не так в коде** (диагностика) и частично **как это исправить** (code_actions).

---

## 3. Разделение ответственности: «что / как / где»

- **Что** (семантика API, контракт, версии):  
  **1c-help** — `get_1c_function_info`, `get_1c_help_topic`, при необходимости `search_1c_help_keyword` + `get_1c_help_topic`.

- **Как** (правильное использование, примеры, типовые сценарии):  
  **1c-help** — `get_1c_code_answer`, `get_1c_help_topic`; при нерелевантности — `search_1c_help_keyword("Тип.Метод")` + `get_1c_help_topic`. После удачного решения — `save_1c_snippet`.

- **Где** (в кодовой базе):  
  **lsp-bsl-bridge** — `project_analysis`, `symbol_explore`, `call_graph`. Для переименований — `call_graph` → `prepare_rename` → `rename`.

Рефакторинг опирается на оба MCP: 1c-help — чтобы не сломать смысл API; lsp-bsl-bridge — чтобы видеть зависимости и проверять код.

---

## 4. Достаточно ли агенту информации для рефакторинга

**Да, достаточно**, если использовать оба MCP по назначению:

1. **Перед рефакторингом**  
   - `call_graph` + `project_analysis` (lsp-bsl-bridge) — границы влияния, места вызовов.  
   - При сомнениях в API: `get_1c_function_info` или `search_1c_help_keyword` + `get_1c_help_topic` (1c-help) — что метод делает и как его правильно вызывать.

2. **Во время рефакторинга**  
   - Правка по одному файлу.  
   - После правки: `document_diagnostics(uri)` (lsp-bsl-bridge).  
   - При необходимости заменить вызов на другой API: `get_1c_code_answer` / `get_1c_function_info` (1c-help).

3. **После правок**  
   - Цикл: исправление → `document_diagnostics` до отсутствия ERROR.  
   - После batch: `did_change_watched_files` (lsp-bsl-bridge).  
   - Удачный новый код → `save_1c_snippet` (1c-help).

Ограничения, о которых агенту нужно помнить (из `quality-and-pitfalls-analysis.md`):

- Частичная индексация: в выдаче только проиндексированные версии/разделы; для точных фактов — `search_1c_help_keyword` + `get_1c_help_topic`.
- В семантику попадают первые 2000 символов топика; для точных имён API — keyword-поиск.
- Имена методов в поиск передавать целиком: `Тип.Метод`.
- В `get_1c_code_answer` контент топика обрезается (MCP_MAX_TOPIC_CHARS); полный текст — только через `get_1c_help_topic(topic_path)`.

---

## 5. Краткий чеклист для агента при рефакторинге

- [ ] Вызвать **call_graph** и **project_analysis** для охвата влияния.
- [ ] При работе с незнакомым API: **get_1c_function_info** или **search_1c_help_keyword** + **get_1c_help_topic**.
- [ ] Для модуля формы при необходимости: **get_module_info**, **get_form_metadata**.
- [ ] После каждой правки: **document_diagnostics(uri)**; при ERROR/WARNING — исправить и повторить.
- [ ] После batch-правок: **did_change_watched_files**.
- [ ] Если введён новый удачный паттерн кода: **save_1c_snippet**.

Итого: информации по MCP 1c-help и lsp-bsl-bridge **достаточно**, чтобы выполнять рефакторинг и понимать, что должно работать, как и где. Важно не ограничиваться одним MCP: 1c-help — для справки и примеров, lsp-bsl-bridge — для структуры проекта и диагностики.
