# Анализ переменных окружения: дублирование и понятность

## 1. Дублирование и алиасы

### 1.1 HELP_SOURCE_BASE

Единственная переменная для корня каталогов с версиями 1С (.hbk). Раньше поддерживался алиас HELP_SOURCES_DIR — удалён.

---

### 1.2 STANDARDS_REPOS

Единственная переменная для списка репозиториев стандартов (через запятую, owner/repo:branch). Раньше поддерживался STANDARDS_REPO (один репо) — удалён; один репо задаётся как STANDARDS_REPOS с одним элементом.

---

### 1.3 REDIS_URL / REDIS_HOST + REDIS_PORT

| Переменная   | Назначение              | Рекомендация |
|--------------|-------------------------|--------------|
| **REDIS_URL** | Полный URL (redis://host:port/db) | Удобно для одного значения |
| **REDIS_HOST** | Хост                    | Альтернатива при разборке URL |
| **REDIS_PORT** | Порт (по умолчанию 6379) | Используется только при REDIS_HOST без REDIS_URL |

**Вывод:** Дублирования нет: два способа задать подключение (URL или HOST+PORT). Оставить оба; в env.example указать: «REDIS_URL или REDIS_HOST и при необходимости REDIS_PORT».

---

### 1.4 HELP_PATH vs HELP_SOURCE_BASE vs DATA_DIR

| Переменная         | Назначение |
|--------------------|------------|
| **HELP_PATH**      | Каталог, откуда MCP читает топики с диска (get_1c_help_topic). Обычно туда кладётся распакованная справка. |
| **HELP_SOURCE_BASE** | Корень, откуда ingest ищет .hbk (подпапки = версии). Источники для индексации. |
| **DATA_DIR**       | Корень данных проекта: BM25, маркеры ingest, по умолчанию от него DATA_UNPACKED_DIR и каталог маркеров. |

**Вывод:** Дублирования нет — три разные роли:

- **HELP_SOURCE_BASE** — «где лежат исходные .hbk»
- **HELP_PATH** — «откуда MCP отдаёт уже распакованные топики»
- **DATA_DIR** — «корень всех рабочих данных приложения»

По умолчанию HELP_PATH = get_data_dir() (т.е. `data`), а распакованная справка часто в DATA_UNPACKED_DIR = data/unpacked; в Docker HELP_PATH=/data и туда маунтится unpacked. Имеет смысл в документации явно разделить: «источники (.hbk)» / «каталог выдачи справки (MCP)» / «корень данных».

---

### 1.5 HELP_SOURCE_BASE vs HELP_SOURCE_DIRS

| Переменная         | Формат / назначение |
|--------------------|----------------------|
| **HELP_SOURCE_BASE** | Один каталог, внутри — подпапки-версии с .hbk. Сканирование по структуре. |
| **HELP_SOURCE_DIRS** | Список путей через запятую: `path1,path2` или `path1:version1,path2:version2`. Явный список. |

**Вывод:** Два способа задать источники для ingest, не дубликаты. Оставить оба; в env.example кратко описать: «HELP_SOURCE_BASE — один корень с версиями; HELP_SOURCE_DIRS — явный список путей (и при необходимости версий)».

---

## 2. Понятность названий

### 2.1 HELP_* и INGEST_*

- **HELP_PATH** — путь к «справке» для MCP (чтение с диска). Название понятное.
- **HELP_SOURCE_BASE** — база источников .hbk. Понятно.
- **HELP_LANGUAGES** — языки справки (ru, en). Понятно.
- **HELP_FILE_ENCODING**, **HELP_HTML_MAX_BYTES** — кодировка и лимит размера HTML. Понятно.
Временный каталог при ingest (INGEST_USE_TEMP=1) задаётся **INGEST_TEMP_DIR**. Раньше поддерживался HELP_INGEST_TEMP — удалён.

---

### 2.2 INGEST_CACHE_FILE

- Сейчас: путь к **файлу**, родительская папка которого используется для маркеров (load_*.running, status). Сам кэш данных — в Redis.
- Название «CACHE_FILE» намекает на файл кэша (например SQLite), тогда как файл нужен только для выбора каталога маркеров.

**Рекомендация:** Не переименовывать (меняет контракт и дефолты). В env.example и README явно писать: «Путь к файлу, чья **родительская папка** — каталог маркеров; данные кэша хранятся в Redis».

---

### 2.3 INDEX_STATUS_* vs INGEST_*

- **INDEX_STATUS_INTERVAL_SEC** — как часто писать статус ингеста в Redis (секунды).
- **INDEX_STATUS_SAMPLE_SIZE** — размер выборки точек для get_index_status (версии/языки).

«Index» здесь — «индексация» (процесс), а не «индекс Qdrant». Чтобы не путать, в документации можно писать «статус индексации / ingest» или рассмотреть в будущем префикс **INGEST_STATUS_** (INGEST_STATUS_INTERVAL_SEC и т.д.) с обратной совместимостью через старые имена.

---

### 2.4 WATCHDOG_INGEST_TIMEOUT

Таймаут одного запуска ingest при работе watchdog. Используется **WATCHDOG_INGEST_TIMEOUT**. Раньше поддерживался INGEST_WATCHDOG_TIMEOUT — удалён.

---

## 3. Что можно объединить или упростить

### 3.1 Не объединять

- **DATA_DIR** и **DATA_UNPACKED_DIR** — разная семантика; возможность задать unpacked отдельно нужна (в т.ч. в Docker).
- **SNIPPETS_DIR** и **SNIPPETS_JSON_PATH** — каталог сниппетов vs один JSON-файл; роли разные.
- **STANDARDS_DIR** — и приём клонированных репо, и источник при отсутствии STANDARDS_REPOS; один параметр с двумя ролями допустим.

### 3.2 Упрощение без слияния переменных

- Один явный **префикс по зонам**: HELP_ (справка), INGEST_ (индексация), DATA_ (данные), MCP_, EMBEDDING_, STANDARDS_, SNIPPETS_, WATCHDOG_, MEMORY_, QDRANT_, REDIS_. Так и сделано; зафиксировано в env.example.

---

## 4. Рекомендации по действиям

### 4.1 Документация (без смены контракта)

1. **env.example / README**
   - Для **HELP_PATH**, **HELP_SOURCE_BASE**, **DATA_DIR** — общая табличка: «источники .hbk» / «каталог справки для MCP» / «корень данных».
   - Для **INGEST_CACHE_FILE** повсеместно писать: «родительская папка = каталог маркеров; кэш — в Redis».
   - Для **INDEX_STATUS_*** в скобках указывать «статус индексации / ingest».

2. **Единый список по префиксам**
   - В начале env.example или в отдельном разделе: краткая таблица префиксов (HELP_, INGEST_, DATA_, MCP_, …) и что они означают.

### 4.2 Legacy удалён

- **HELP_SOURCES_DIR** — поддержка убрана из `get_help_source_base()`, используется только HELP_SOURCE_BASE.
- **STANDARDS_REPO** — поддержка убрана из cmd_load_standards, используется только STANDARDS_REPOS (один репо задаётся списком из одного элемента).
- **HELP_INGEST_TEMP** — убрана, используется только INGEST_TEMP_DIR.
- **INGEST_WATCHDOG_TIMEOUT** — убрана, используется только WATCHDOG_INGEST_TIMEOUT.
