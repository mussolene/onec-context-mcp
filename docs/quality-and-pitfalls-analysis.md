# Качество ответов MCP 1c-help: анализ причин и подводные камни

Документ описывает, что может влиять на качество ответов инструментов 1c-help, как получать готовый код и чего избегать. Актуален при частичной индексации и при типичных запросах (реквизиты формы, события, версии API).

---

## 1. Может ли влиять незавершённая индексация

**Да.** Пока индексация не закончена, на качество влияет следующее.

### Что уже в индексе — то и ищется

- **Поиск** (semantic + keyword) возвращает только точки из Qdrant. Если нужная версия справки (например 8.2) или раздел ещё не проиндексированы, по ним не будет ни одного результата.
- **get_topic_content** сначала пытается прочитать топик с диска (`HELP_PATH`), затем из payload Qdrant. Если топик не в индексе и файла по `HELP_PATH` нет (типично в Docker без тома с распакованной справкой), контент по такому пути получить нельзя.

### Как проверить, что проиндексировано

- Вызвать **get_1c_help_index_status**: число точек, список версий и языков из payload (по выборке).
- Если нужна версия 8.2 — в выдаче должны быть версии вроде `8.2.x`. Если там только 8.3/8.5 — запросы с фильтром `version="8.2"` будут пустыми или скудными.

### Итог

При индексации «больше чем наполовину» часть справки уже доступна, но:
- ответы по ещё не проиндексированным разделам/версиям будут пустыми или нерелевантными;
- запросы в духе «доступно ли в 8.2» могут не находить статей именно 8.2, если эта версия в индексе слабо представлена.

После полного завершения ingest качество по «покрытым» версиям и разделам стабилизируется; улучшится и ответ на вопросы по старым версиям (8.2 и т.д.), если они реально попали в индекс.

## 2. Технические причины просадок качества

### 2.1. Обрезка текста при эмбеддинге (2000 символов)

- При индексации в вектор идёт **первые 2000 символов** каждого документа (`MAX_EMBEDDING_INPUT_CHARS` в `embedding.py`). Остальной текст хранится в payload (до 50 000 символов) и отдаётся при `get_1c_help_topic`, но **не участвует в семантическом поиске**.
- Если ключевая формулировка («реквизит формы», «ПриОткрытии») находится после первых 2000 символов страницы, эта страница может слабо ранжироваться по семантике или вообще не попадать в топ.

**Что делать:** для точных терминов и имён API опираться на **search_1c_help_keyword** (BM25/подстрока по заголовку и тексту), а не только на `get_1c_code_answer`.

### 2.2. Запрос при поиске (семантика)

- Для **local** (sentence-transformers) запрос в эмбеддер передаётся **целиком** (без обрезки).
- Для **openai_api** запрос обрезается до **2000 символов** перед отправкой в API.

Длинные запросы при `EMBEDDING_BACKEND=openai_api` могут терять часть смысла из-за обрезки.

### 2.3. Извлечение ключевых слов из запроса

В hybrid-поиске (`get_1c_code_answer`) для keyword-фазы из запроса извлекаются:

- шаблон **Тип.Метод** (например `УправляемаяФорма.ПриОткрытии`);
- отдельные слова в CamelCase/кириллице длиной ≥ 3 символов (до 8 токенов).

Фразы вроде «добавить реквизит на форму» дают токены «добавить», «реквизит», «форму» и т.п. Поиск по ним идёт по подстроке/BM25. Если в справке нет страницы, где всё это собрано в одном месте (типично для методических сценариев), топ может быть слабым или нерелевантным.

### 2.4. Нет «одной страницы» под сценарий

Справка 1С описывает объекты и методы, а не пошаговые сценарии («добавить реквизит → вывести на форму → обработчик события»). Поэтому:

- **get_1c_code_answer** по такому запросу часто тянет разрозненные страницы (декорации, картинки, виды изменений и т.д.);
- готовый код по сложному сценарию чаще собирается из нескольких топиков (реквизит формы, события формы, элементы) или из памяти (сниппеты), а не из одной статьи.

### 2.5. Контент топика при выдаче

- В **get_1c_code_answer** и **search_1c_help_with_content** контент каждого топика обрезается до **MCP_MAX_TOPIC_CHARS** (по умолчанию 4000). Полный текст — только через **get_1c_help_topic(topic_path)**.
- Если ответ «не дотягивает», имеет смысл взять `path` из выдачи и вызвать **get_1c_help_topic** без ограничения по длине (если топик есть в индексе или на диске).

### 2.6. Одна модель при индексации и при поиске

Если при ingest использовался один бэкенд/модель эмбеддингов, а при работе MCP — другой (или другая размерность), семантический поиск даёт нерелевантные результаты. В Docker для `mcp` и `ingest-worker` должны совпадать `EMBEDDING_BACKEND`, `EMBEDDING_MODEL`, при необходимости `EMBEDDING_DIMENSION` (см. `docs/embedding.md`).

## 3. Проблемные места по типам запросов

| Тип запроса | Типичная проблема | Рекомендация |
|-------------|-------------------|--------------|
| Готовый код по сценарию (реквизит на форму + обработчик) | В справке нет одной страницы под сценарий; семантика тянет общие темы. | Разбить на конкретные API: `search_1c_help_keyword("реквизит формы")`, `search_1c_help_keyword("ПриОткрытии")`, `get_1c_help_topic` по нужным path; при необходимости дополнять из памяти (сниппеты). |
| «Как правильно писать код» / стандарты | В справке по API такого раздела может не быть; поиск даёт «Код» (поле), запросы и т.п. | Опираться на внешние стандарты (EDT, BSL LS, v8-code-style) и **load-standards**; не ожидать от справки полного гайда по стилю. |
| «С какой версии доступно X» | Хорошо работает **search_1c_help_keyword** по названию X (например «интерактивный режим криптографии»). | Искать точное имя фичи или метода; смотреть блок «Использование в версии» в топике. |
| «Доступно ли в 8.2» (конкретное свойство) | В индексе может не быть или быть мало статей именно 8.2; путь может быть с префиксом версии. | Проверить **get_1c_help_index_status** (есть ли версия 8.2); искать по ключевому слову и при необходимости переходить к **get_1c_help_topic** по path из другой версии, сверяя текст. |

## 4. Как получать готовый код и знать подводные камни

### Рекомендуемый порядок вызовов (кратко)

1. **get_1c_code_answer(query)** — первый вызов для кода и примеров.
2. Если результат нерелевантен или мал:
   - **search_1c_help_keyword(точное_имя_API)** (например `УправляемаяФорма.ПриОткрытии`, `МенеджерКриптографии`);
   - затем **get_1c_help_topic(topic_path)** по `path` из выдачи.
3. Для методов — передавать полное имя: **Тип.Метод** (например `HTTPСоединение.Получить`).
4. После получения рабочего кода — **save_1c_snippet**, чтобы улучшить последующие ответы.

### Известные подводные камни (уже в описаниях инструментов)

- **ПрочитатьJSON** по умолчанию возвращает **Структуру**; для **Соответствия** нужно указывать `ПрочитатьВСоответствие=Истина`.
- **HTTPСоединение.Получить** — только на сервере.
- Имена методов в **search_1c_help_keyword** и **get_1c_function_info** передавать целиком: `Тип.Метод`.

### Что проверить при плохих ответах

1. **get_1c_help_index_status** — не пустой ли индекс, какие версии/языки есть.
2. При необходимости — фильтры **version** и **language** в поиске.
3. Повторить запрос через **search_1c_help_keyword** с точным именем API или объекта.
4. Убедиться, что бэкенд/модель эмбеддингов при ingest и при работе MCP совпадают.

## 5. Влияние индексации: итог

- **Частичная индексация** напрямую ограничивает: (1) какие пути вообще попадают в выдачу поиска; (2) по каким путям можно прочитать контент из Qdrant (если с диска по `HELP_PATH` читать нечего).
- После завершения ingest по уже проиндексированным версиям и разделам качество стабилизируется; запросы по версиям, которых в индексе нет или мало, по-прежнему будут давать пустоту или слабые результаты.
- Для максимальной предсказуемости: дождаться окончания индексации, проверить **get_1c_help_index_status**, и для точных фактов (версии, имена API) предпочитать **search_1c_help_keyword** + **get_1c_help_topic**.
