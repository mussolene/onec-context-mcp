## 2. Результаты тестовых запросов

### 2.1 Запрос: `ПрочитатьJSON` (точное имя API)

| Метод | Результат |
|-------|-----------|
| `search_1c_help` | «Построитель отчета» — нерелевантно |
| `search_1c_help_keyword` | ТипУзораТабличногоДокумента — нерелевантно |
| `get_1c_function_info` | «No exact match» — корректно указывает на search_1c_help_keyword |
| `get_1c_code_answer` | ТипУзораТабличногоДокумента — нерелевантно |

**Вывод:** Функция `ПрочитатьJSON` не находится ни семантикой, ни keyword. Возможные причины: путь в индексе другой (zif/ru) или BM25/семантика промахиваются.

### 2.2 Запрос: `HTTPСоединение.Получить`

| Метод | Результат |
|-------|-----------|
| `search_1c_help_keyword` | КатегорияИспользованияАтрибутаXS.Обязательно, РольПоляНабораДанных...Обязательное — нерелевантно |

**Вывод:** Стемминг «Получить» даёт совпадения с «Обязательно/Обязательное» и др. Ранжирование BM25 выводит наверх не те документы.

### 2.3 Запрос: `Формат` (глобальная функция)

| Метод | Результат |
|-------|-----------|
| `search_1c_help_keyword` | Расширения формы (ОтображатьПрогнозируемыеКолонки и т.п.) — частично релевантно |
| `get_1c_function_info` | ФорматШапки (свойство) — не целевая функция Формат |

**Вывод:** Находятся свойства/типы с «Формат» в названии, глобальная функция `Формат` — не на первом месте.

### 2.4 Запрос: `вывод СКД в таблицу` (описательный)

| Метод | Результат |
|-------|-----------|
| `search_1c_help` | «Сводная таблица» — частично релевантно (таблица есть, но не вывод СКД) |

**Вывод:** Семантика даёт связанную тему, но не точный ответ по выводу СКД.

### 2.5 Запрос: `Формат числа`

| Метод | Результат |
|-------|-----------|
| `get_1c_code_answer` (code_only=True) | Расширения табличного поля модели прогноза — нерелевантно |

**Вывод:** Hybrid выбирает не ту тему; глобальная функция Формат не выходит в топ.

## 3. Что стало лучше

1. **BM25 по умолчанию** — keyword-поиск использует sparse vectors вместо только payload.keywords.
2. **Изоляция vocab в тестах** — `data/bm25_vocab` больше не перезаписывается при прогоне тестов.
3. **Лимиты контента** — MCP_MAX_TOPIC_CHARS=4000, MCP_SNIPPET_MAX_CHARS=1200 настраиваются через env.
4. **Подсказки в docstring** — явные советы по `search_1c_help_keyword` и `Тип.Метод`.
5. **Подсказка при низком score** — «При нерелевантных результатах попробуйте search_1c_help_keyword».

---

## 4. Что остаётся проблемой

1. **Точные имена API** — `ПрочитатьJSON`, `Формат`, `HTTPСоединение.Получить` часто не в топе или не находятся.
2. **Стемминг BM25** — «Получить» даёт лишние совпадения; для `Тип.Метод` нужна приоритизация точного совпадения.
3. **Ранжирование** — при многих совпадениях по общим словам релевантный документ теряется.
4. **Пути справки** — функции из `zif/` (запросы, общие) могут иметь другую структуру path; нужна проверка индекса.
5. **Описательные запросы** — «вывод СКД в таблицу» не даёт точного ответа по выводу СКД в таблицу.

---

## 5. Рекомендации

### 5.1 Краткосрочные

1. **Проверить индексацию zif** — есть ли `ПрочитатьJSON` и подобные в индексе, под какими path.
2. **Буст точного совпадения в заголовке** — при `Type.Method` или точном имени в title повышать score.
3. **Уточнить docstring get_1c_function_info** — при «No exact match» предлагать `search_1c_help_keyword` с полным именем.

### 5.2 Среднесрочные

1. **Гибридный ранкер** — комбинировать semantic + BM25 score (например, RRF) вместо простого «keyword first».
2. **Фильтр по типу контента** — для `get_1c_function_info` сужать поиск до «функции/методы» при наличии такого payload.
3. **Синонимы API** — таблица частых синонимов (ПакетПолучения → ВыполнитьПакет) в skill; при необходимости — в коде.

### 5.3 Связка с BSL LS / lsp-bsl-bridge

- **1c-help** — справка, примеры, поиск по API.
- **lsp-bsl-bridge** — статический анализ, diagnostics, code actions.

Рекомендуемый порядок: `get_1c_code_answer` → код → `document_diagnostics` → исправления.

---

## 6. Заключение

Текущий подход (BM25 + семантика + память) даёт рабочий поиск, но для **точных имён API** качество недостаточное: стемминг и ранжирование часто выводят нерелевантные документы. Для описательных запросов семантика работает приемлемо.

Основные направления улучшения:
1. Усиление точного совпадения по заголовку при keyword-поиске.
2. Проверка покрытия индекса (zif, общие функции).
3. Улучшение hybrid-ранжирования (RRF или взвешенная комбинация scores).

## 7. Внедрённые улучшения (после анализа)

### Title boost в search_index_keyword

Добавлена пересортировка результатов BM25: документы, в заголовке которых содержится точная подстрока запроса (`query in title`), выводятся первыми.

**Эффект:** для запросов вроде «ПрочитатьJSON», «ФоновоеЗаданиеКлиентСервер», «HTTPСоединение.Получить» релевантная тема (если она есть в индексе и в title) поднимается в топ.

**Изменения:** `src/onec_help/indexer.py` — 4 строки перед `return out` в BM25-ветке.
