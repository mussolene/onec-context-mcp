# Отладка 1С, доступ к СУБД и инкрементальная работа с конфигурацией (XML)

Краткий справочник: как **не путать** SSH-шлюз автономного сервера, доступ к **базе данных** и **инкрементальную выгрузку/загрузку метаданных**. Синхронизировано с skill [1c-platform-cli](../cursor-examples/1c-platform-cli/SKILL.md).

## 1. Три разных «SSH»

### 1.1. SSH-шлюз `ibsrv` (1С:Предприятие)

- Это **протокол администрирования** автономного сервера: **`ibcmd --remote ssh://…`**, конфигуратор в агентском режиме, SFTP в пользовательский каталог по документации.
- **Не** является:
  - доступом к PostgreSQL / MS SQL по wire protocol;
  - заменой **сервера отладки**;
  - универсальной оболочкой `ssh user@host` с shell (обычный OpenSSH-клиент к этому порту не даёт интерактивной сессии 1С).

### 1.2. OpenSSH port forwarding (классический туннель)

- Пример: `ssh -L 15432:db.internal:5432 jump.example.com` — на ноутбуке поднимается `localhost:15432`, который пробрасывается на **порт СУБД** за границей.
- Строка подключения 1С (клиент-сервер) настраивается на **localhost:15432** (или иной локальный порт), если политика безопасности так допускает.
- Это **стандартный сетевой** приём, не спецификация платформы 1С.

### 1.3. VPN или прямой доступ в корпоративной сети

- Для ежедневной разработки часто **проще** постоянный VPN до сегмента, где доступны **менеджер кластера**, **рабочие процессы** и при необходимости **СУБД**, чем отдельный туннель под каждый инструмент.

## 2. Отладка кода (BSL)

- Отладчик подключается к **серверу отладки** платформы (отдельный контур от SSH-шлюза `ibsrv`).
- У **автономного сервера** (`ibsrv`) в справке задаются параметры вроде **`--debug`**, **`--debug-port`**, **`--debug-address`**, **`--debug-password`** — см. приложение администратора по вашей версии (например [Appendix 4.9 — ibsrv](https://kb.1ci.com/1C_Enterprise_Platform/Guides/Administrator_Guides/1C_Enterprise_8.3.27_Administrator_Guide/Appendix_4._Startup_command_lines_of_system_components_and_description_of_additional_utilities/4.9._Standalone_server__ibsrv_/?language=en), подставьте актуальную подверсию в URL).
- В **клиент-серверном** режиме сценарий отладки описан в руководстве администратора/разработчика по **отладке** и сетевой доступности портов между ПК разработчика и серверами приложений.
- **Файловая ИБ:** отладка обычно ведётся **локально**; удалённый доступ к каталогу базы — через **сетевую папку**, **SSHFS**, копирование и т.п., это уже уровень ОС, не «SSH к базе» в терминах 1С.

**Вывод:** для отладки не нужно «изобретать» доступ через SSH-шлюз `ibsrv` к СУБД; нужны **корректная публикация отладки** и **сеть** (VPN/туннель при необходимости).

## 3. Инкрементальные изменения конфигурации (XML)

Речь о **метаданных** (выгрузка конфигурации в файлы), а не об инкрементальном обновлении строк в таблицах SQL.

### 3.1. `ibcmd config export` (8.5.x, сверяйте `ibcmd help config`)

- **`--base` / `-b`** — путь к **ConfigDumpInfo**: «относительно которого вычисляются изменения текущей конфигурации».
- **`--sync`** — синхронизировать уже существующую выгрузку в каталоге с текущей конфигурацией.
- **`--force`** — выполнить полную выгрузку, если версия формата выгрузки не совпадает с записанной в `ConfigDumpInfo.xml`.
- **`--threads`** — параллелизм **внутри** одной операции экспорта, не замена нескольких независимых заданий на одной ИБ.

### 3.2. `ibcmd config import files`

- **`--partial`** — разрешить **частичный** набор файлов объектов метаданных. Требует дисциплины: неполный набор может нарушить ссылки и целостность, если регламент выгрузки/слияния не выдержан.

### 3.3. EDT, хранилище, Git

- В типичной разработке **инкремент** живёт в **модели EDT** и **системе контроля версий** (или хранилище конфигурации), а не в постоянном цикле «полный `ibcmd` на каждый чих».

## 4. Будет ли «нормально работать»

| Условие | Комментарий |
|---------|-------------|
| Одна **версия платформы** на контуре выгрузки/загрузки | Снижает сюрпризы формата XML |
| Регламент по **ConfigDumpInfo** и полным выгрузкам при смене формата | Иначе `--sync` и сравнение изменений дают шум или требуют `--force` |
| **Частичный импорт** только по согласованному процессу | Иначе выше риск поломки метаданных |
| Отладка: доступен **debug**-контур и ИБ по сети | SSH к `ibsrv` для админки и отладка — разные оси |
