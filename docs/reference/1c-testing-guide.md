# Тестирование кода 1С: где, как и что использовать

Читайте этот файл, когда нужно понять, чем проверять Python-часть проекта и чем тестировать 1С-код и сценарии.

Справочник для ассистента и разработчиков: какие фреймворки применять для тестирования в среде 1С, где искать тесты и как их запускать.

## Два основных направления

| Направление | Фреймворк | Назначение | Где обычно лежат тесты |
|-------------|-----------|------------|------------------------|
| **Unit / модульные тесты** | **YaxUnit** (или xUnitFor1C) | Тестирование процедур и функций BSL изолированно (моки, стабы). | В репозитории конфигурации 1С: `Tests/`, подсистемы тестов, внешние обработки с тестами. |
| **BDD / сценарии / UI** | **Vanessa-Automation** (xdd, UI) | Сценарии на Gherkin (.feature), data-driven (xdd), автоматизация UI и приёмочные тесты. | В репо конфигурации или отдельном репо: `features/`, `BDD/`, каталоги с `.feature` и шагами. |

---

## 1. YaxUnit (unit-тесты)

- **Что это:** фреймворк unit-тестов для 1С:Enterprise (BIA Technologies, открытый). Запуск тестов в 1С:Предприятие или из EDT.
- **Когда применять:** проверка логики процедур/функций, граничных условий, обработки ошибок без поднятия полного сценария.
- **Где искать:** в проекте конфигурации 1С — подсистемы с тестами, папки `Tests/`, модули с префиксом или суффиксом «Тест», внешние обработки/расширения с тестами.
- **Как запускать:** через 1С:Предприятие (форма тестов), EDT (плагин YaxUnit), или CLI/OneScript, если настроен.
- **Для ассистента:** при добавлении новой логики в модули 1С предлагать написать unit-тест (YaxUnit); искать существующие тесты в `Tests/`, `*Тест*`, подсистемах тестирования.

## 2. Vanessa-Automation (BDD, xdd, UI)

- **Что это:** BDD-фреймворк для 1С (Gherkin), плюс xdd (data-driven сценарии) и автоматизация UI.
- **Когда применять:** приёмочные сценарии, интеграционные проверки, сценарии с формами и UI, проверки по данным (xdd).
- **Где искать:** в репозитории конфигурации или в отдельном репо с тестами — каталоги `features/`, `BDD/`, `Vanessa/`, файлы `*.feature`, шаги (Steps).
- **Как запускать:** через 1С:Предприятие с подгруженной Vanessa-Automation, запуск набора .feature; часто есть скрипты в корне репо или в `tools/`.
- **Для ассистента:** при добавлении сценариев или изменении поведения по сценариям искать/редактировать `.feature` и шаги; не путать с unit-тестами — здесь уровень «сценарий», а не одна процедура.

## Роль MCP при тестировании 1С

MCP уже пригоден при написании и проверке тестов 1С; отдельный MCP для *запуска* тестов возможен как развитие.

| MCP | Как пригодиться при тестировании |
|-----|----------------------------------|
| **1c-help** | При написании тестов: `get_1c_code_answer` — примеры вызовов API для тестируемого кода; `search_1c_help_keyword` — точная сигнатура типа/метода; `save_1c_snippet` — сохранить удачный тестовый пример в память. Полезен и для кода теста (YaxUnit, шаги Vanessa), и для тестируемого кода. |
| **lsp-bsl-bridge** | Проверка кода тестов: `document_diagnostics(uri)` — ошибки и предупреждения BSL LS в модулях тестов и шагах (.bsl). После правок теста вызывать diagnostics до чистоты. `project_analysis`, `symbol_explore` — навигация по тестам и тестируемому коду. |
| **Запуск тестов (потенциально)** | Сейчас YaxUnit и Vanessa запускаются вручную (1С:Предприятие, EDT, скрипты). Отдельный MCP с инструментами вида `run_yaxunit`, `run_vanessa_features` мог бы вызывать OneScript/EDT/CLI и возвращать результат в чат — такой MCP в этом репо не реализован; при появлении его стоит описать здесь. |

**Для ассистента:** при предложении или правке тестов 1С использовать 1c-help для примеров API и lsp-bsl-bridge для проверки кода теста (`document_diagnostics` по URI модуля теста/шагов).

## В рамках текущего репозитория (1c_hbk_helper)

- Этот репозиторий — **Python-проект** (справка 1С, MCP, индексация). Тесты **кода 1С** здесь не хранятся.
- **Python-тесты:** `tests/`, pytest, покрытие ≥70%. Команды — в AGENTS.md и в правиле `1c-testing-workflow.mdc`.
- Выгрузка конфигурации 1С для графа метаданных (`data/config` или ONEC_CONFIG_SOURCE_DIR) относится к NDA‑периметру и не должна копироваться в публичные репозитории или артефакты CI.
- **Справочная информация по тестам 1С** — этот документ и правила в `docs/cursor-examples/` нужны, чтобы ассистент при работе с кодом 1С (в другом репозитории или в связанном проекте) понимал, что и как тестировать.

## За рамками репозитория (репо конфигурации 1С)

- Тесты 1С (YaxUnit, Vanessa) живут в **репозитории конфигурации 1С** или в отдельном репо с тестами.
- Рекомендуется в том репо:
  - **Правила Cursor:** скопировать или адаптировать `docs/cursor-examples/rules/1c-testing-workflow.mdc`, указать фактические пути к `Tests/`, `features/`, BDD.
  - **README или CONTRIBUTING:** описать, как запускать YaxUnit и Vanessa (команды, EDT, скрипты).
  - **Структура каталогов:** явно выделить, например, `Tests/` (YaxUnit), `features/` или `BDD/` (Vanessa), чтобы ассистент знал, где искать.

## Краткая памятка для ассистента

1. **Уточнить контекст:** код 1С пишется в этом репо (только справка/MCP) или в репо конфигурации? Тесты 1С — только в репо конфигурации.
2. **Unit-логика (процедуры/функции):** предлагать тесты **YaxUnit** (или xUnitFor1C); искать тесты в `Tests/`, подсистемах тестов.
3. **Сценарии, UI, приёмка:** предлагать/править **Vanessa-Automation** (`.feature`, xdd, UI); искать в `features/`, `BDD/`.
4. **После правок BSL:** всегда предлагать `document_diagnostics` (BSL LS); при новой логике — предложить unit (YaxUnit) или сценарий (Vanessa) в зависимости от уровня проверки.
